# 如何为 AI 构建工具：从 trace-ui 到 trace-cli 的重构实践

> 本文记录了我们将一个传统逆向分析 GUI 工具重构为 AI Agent 专用 CLI 的完整过程。不是技术教程，而是一次产品思维的转变记录——在 AI 时代，工具应该怎么设计？

## 背景

我们有一个 ARM64 trace 分析工具 [trace-ui](https://github.com/imj01y/trace-ui)，是一个 Tauri + React 的桌面应用。它做的事情很直接：把 unidbg 跑出来的几十万、几百万行指令级 trace 文件，变成人能看懂的东西——虚拟滚动浏览、调用树折叠、反向污点追踪、内存面板、寄存器面板……

工具本身没问题。但我们想让 AI 来辅助分析这些 trace，问题就来了：**一个为人类设计的 GUI 工具，AI 用不了。**

于是我们开始重构。以为很简单——加个 CLI 接口，把 GUI 的能力暴露出来就行。结果发现，这不是加接口的问题，而是整个产品思维需要推倒重来。

## 第一个坑：把 AI 当人用

最初的想法很自然：trace-ui 有调用树、有污点分析、有内存面板、有字符串提取、有密码学扫描……给 AI 也提供这些功能不就行了？

于是我们设计了一大堆命令：

```
overview    —— 全局函数树
func        —— 单个函数详情
lines       —— 查看指令
search      —— 搜索
taint       —— 污点分析
deptree     —— 依赖树
calltree    —— 调用树展开
strings     —— 字符串提取
crypto      —— 密码学常量扫描
memory      —— 内存查询
xref        —— 交叉引用
memdump     —— 内存重构
```

12 个命令。看起来很全面。

**问题是：AI 不需要这么多。**

密码学常量扫描？AI 看到 `0x67452301` 自己就知道那是 MD5 IV。字符串检测？AI 看到 `mem[WRITE]` 里的 ASCII 字节自己能认出来。依赖树表达式还原？AI 从污点分析的指令列表自己能推理出计算逻辑。

这些功能本质上是在**替 AI 思考**。传统工具需要这些，是因为人的认知带宽有限，需要工具预处理信息。但 AI 的认知带宽很大，它缺的不是预处理后的结论，而是**高效访问原始信息的能力**。

我们砍掉了一半以上的命令。

## 第二个坑：给 AI 画地图

砍完命令后，我们觉得 AI 总得知道"该看哪里"吧？于是设计了一个 `overview` 命令——把整个 trace 的函数调用树展示出来，像目录页一样。

```
Trace: libxyass.so  139402 lines  unidbg

0x1209e184  139401 insns
├─ 0x4fe294    152 insns   x12
├─ 0x197da4    204 insns   x8   loop:10
├─ 0x189ba4    125 insns   x4   loop:16
└─ ...
```

还特意加了"循环检测"和"调用次数"标注，让 AI 快速定位关键函数。

用了一下就发现：**对混淆代码完全没用。**

libxyass.so 是重度混淆的——函数调用被打散成 10 层单子函数套娃，每层只有一个 child。overview 显示的函数树看起来是：

```
0x1209e184 → 0x1206e3a0 → 0x1206e13c → 0x1209e48c → 0x120a6a30 → ...
```

AI 看到这个，和没看一样。要手动钻 10 层才能到真正的逻辑。

而 `search "0x67452301"` 一秒钟就定位到了 MD5 IV 在哪。`search "movi v2.16b, #0x36"` 直接找到了 HMAC 的 ipad XOR。

**AI 不需要地图。AI 需要搜索。**

给 AI 画地图，是在用人类的导航方式强加给 AI。人类需要先建立全局认知再深入细节。AI 不需要——它可以直接搜索感兴趣的模式，从任意点开始探索。

我们删掉了 `overview` 和 `calltree`。

## 第三个坑：替 AI 做判断

最开始，我们还设计了"函数行为摘要"，给每个函数自动标注"疑似 AES"、"疑似 HMAC"：

```
func 0x189ba4: 疑似 AES-128 加密
  证据:
  - 16 轮循环
  - 256B 查找表（匹配 AES S-Box）
  - 输入 16B，输出 16B
```

这看起来很智能，但本质上是用**规则匹配替代了 AI 的推理**。规则能覆盖的就那几种已知算法，遇到魔改或自定义算法就完全失效。而 AI 只要看到原始数据——16 轮循环、256 字节表、EOR 操作——自己就能推理出这是什么。

更糟糕的是，错误的标注会误导 AI。如果规则把一个魔改 MD5 标注为"标准 MD5"，AI 可能直接信了，不去深入看魔改点。

**工具不应该做判断，工具应该做计算。**

AI 能推理但不能计算（比如不能在百万行 trace 上跑 BFS 依赖分析）。工具能计算但不应该推理（不应该判断"这是什么算法"）。各司其职。

## 第四个坑：输出太多

最初 taint（污点分析）的输出是这样的：标记了 36,974 行（26% 的 trace），然后输出前 50 行给 AI 看。

前 50 行全是栈初始化和地址计算：

```
sub sp, sp, #0x190
stp x29, x30, [sp, #0x130]
add x29, sp, #0x130
mrs x8, tpidr_el0
...
```

AI 看了 50 行，全是噪声，一点核心算法逻辑都没有。

问题在于 taint 追踪了**所有依赖**——包括控制流依赖（条件分支）和地址计算依赖（栈指针操作）。这些对人工分析可能有用，但对 AI 来说就是噪声。

加了 `--data-only` flag 后：36,974 → 199 行。**99.5% 的噪声被消除。** 前 50 行全是 HMAC key 加载、ipad XOR、MD5 状态更新——全是核心逻辑。

**AI 的上下文窗口就是它的"注意力"。你给它 26% 的 trace 当噪声，就是在浪费它最宝贵的资源。**

"一口一口吃"不只是输出行数的限制，更是信息密度的要求。每一口都应该有营养。

## 第五个坑：假设 AI 知道流程

最初我们设计了一个 `index` 命令——AI 要先跑一次索引，建好缓存，然后再查询。

```
trace-cli index file.txt      # 第一步
trace-cli search "xxx"         # 第二步
```

这是典型的**程序员思维**：先初始化，再使用。

但 AI 不关心你的内部流程。AI 只知道"我要搜索这个 trace 里的某个模式"。它不应该需要知道索引这个概念。

改成了：任何命令进来，自动检查缓存→没有就建→有就用。对 AI 完全透明。

```
trace-cli file.txt search "xxx"    # 就这一步，其他都是后台
```

**对于 AI 产品，用户调用一个"吃饭函数"就能吃到饭，不需要知道怎么烧水做饭。**

## 第六个坑：忽略了工具链上游

trace-cli 本身做得再好，如果 unidbg 输出的 trace 数据不够，一切白搭。

标准 unidbg 的 trace 输出：

```
ldr x8, [x26, #0x50]  x26=0x123db000 => x8=0x60957206
```

没有内存地址。你知道 x8 变成了 `0x60957206`，但不知道这个值是从内存的哪个地址读的。这意味着：
- taint 无法追踪通过内存传递的数据流
- memdump 无法重构内存状态
- xref 无法查找谁读写了某个地址

我们修改了 unidbg 的 `AssemblyCodeDumper.java`，给每条 load/store 加上 `; mem[READ/WRITE] abs=0x...` 注解。

然后遇到了更深的坑：**SIMD 寄存器值读不出来。**

`str q0, [addr]` 这种 128-bit 向量存储，在 CodeHook 回调里通过 `reg_read_vector()` 读到的值全是零。Unicorn2 和 Hypervisor 两个 backend 都一样。

最终的解决方案不是修 `reg_read_vector`（那是模拟器的深层 bug），而是**绕过寄存器，直接读内存**：在 store 指令执行后的下一条指令的 hook 里，用 `mem_read()` 读取刚写入的内存值。

```
str q0, [x27]  ; mem[WRITE] abs=0x123d6070 ... data[0x123d6070]=0x6bc31d303a945570fb1343cbc6dc449f
```

这个 `data[0x...]` 字段就是从内存读出的真值。用它提取出了完整的 64 字节 HMAC key。

**工具不是孤立的。整条链路（模拟器→trace→分析工具→AI）的每个环节都可能成为瓶颈。**

## 最终形态：6 个命令

折腾一圈下来，从 12 个命令砍到 6 个：

```bash
trace-cli <file> info                                    # 基本信息
trace-cli <file> lines <range>                           # 看指令
trace-cli <file> search <pattern> [--range start-end]    # 搜索
trace-cli <file> taint <spec> [--data-only] [--range]    # 追数据流
trace-cli <file> memdump <addr> <size> [--at <seq>]      # 提取内存
trace-cli <file> xref <addr>                             # 内存引用
```

每个命令对应 AI 的一种基本能力需求：
- **info** → 知道规模
- **search** → 找到位置
- **lines** → 看到内容
- **taint** → 理解依赖（唯一不可替代的计算能力）
- **memdump** → 提取数据
- **xref** → 追踪引用

实战验证：用这 6 个命令分析小红书 `libxyass.so` 的 shield 算法（13.9 万行，重度混淆），约 10 次调用识别出完整算法结构（HMAC-MD5 → RC4 → Base64），memdump 提取了完整的 64 字节 HMAC key。

## AI 时代构建工具的几个原则

回头看这次重构，总结几条教训：

**1. 不要替 AI 思考**

传统工具需要"智能"——模式匹配、启发式分析、自动分类。AI 时代的工具应该"笨"——只做计算，把推理留给 AI。你的规则覆盖不了 AI 的推理能力，但你的规则的错误会污染 AI 的判断。

**2. 不要给 AI 设计流程**

传统产品设计：PRD → 用户故事 → 交互流程 → 界面设计。AI 工具设计：定义能力 → 确保每个能力独立可用 → 让 AI 自己决定调用顺序。AI 不需要 workflow，AI 需要 toolbox。

**3. 控制信息密度，不是信息量**

AI 的上下文窗口是固定的。你给 1000 行噪声不如给 50 行精华。`--data-only` 把 26% 的 taint 结果砍到 0.14%，不是丢了信息，是去掉了噪声。

**4. 大模型进化太快，复杂规则会成为负债**

今天 AI 不认识的模式，下个版本可能就认识了。你写了一套复杂的"算法识别规则"，三个月后 AI 自己就能做得更好，而你的规则反而成了维护负担和误导源。**工具应该提供数据访问能力，而不是数据解读能力。**

**5. 关注整条链路，而不仅是工具本身**

我们花了大量时间修 unidbg 的 SIMD 寄存器输出问题。如果只盯着 trace-cli 本身，永远提取不到 HMAC key。工具链的每个环节都可能是瓶颈。

**6. 少即是多**

从 12 个命令砍到 6 个，删掉了 267 行代码，工具反而更好用了。每删掉一个不必要的命令，就减少了 AI 的认知负担和选择成本。**AI 不需要你给它很多选择，它需要每个选择都有效。**

## 最大的认知转变：从需求原型到 Demo 原型

传统产品开发：需求分析 → PRD → 原型设计 → 开发 → 测试。

这套流程的前提是**你能在动手之前想清楚需求**。对 AI 产品来说，这个前提不成立。

我们最初也试过"先想清楚"。花了大量时间设计接口、讨论 Layer 2/Layer 3 摘要格式、规划 overview 的信息密度、设计函数行为标签……写了一份很完整的设计方案，还让另一个 AI 做了评审。

然后真正写出来一跑：overview 对混淆代码没用，calltree 钻 10 层才到逻辑，taint 噪声 26%，memdump 提取不到 SIMD 数据。**设计方案里想得越多，实际验证时推翻得越多。**

真正有效的过程是：

```
写一个最小 demo（3 个命令）
  → 拿真实 trace 让 AI 用
    → 发现 search 最有用，overview 没用
      → 砍掉 overview，加 --range
        → 发现 taint 噪声太大
          → 加 --data-only，36974 → 199 行
            → 发现 memdump 提取不到 SIMD 数据
              → 改 unidbg 的 RegAccessPrinter
                → 发现 reg_read_vector 返回全零
                  → 改成延迟内存读取
                    → HMAC key 完整提取
```

每一步都是"试了才知道"，不是"想了就知道"。

**这就是 AI 产品的核心方法论转变：从需求原型到 Demo 原型。**

说到底，原因只有一个：**我们对 AI 并不完全了解。**

传统产品的用户是人。人的认知模式、操作习惯、能力边界，产品经理可以通过调研、访谈、同理心来把握。你知道人看不了几百万行文本，所以你做虚拟滚动。你知道人记不住调用关系，所以你做调用树。你知道人追踪数据流很慢，所以你做污点分析。需求可以被预先定义，因为你足够了解你的用户。

但 AI 不一样。

AI 能识别什么模式？看到 `0x67452301` 能不能自己认出 MD5？需不需要你标注？看到 16 轮循环能不能推断出 AES？一次给它 50 行够不够？200 行会不会太多？它追踪数据流的思路和人一样吗？

**你不知道。在你真正让 AI 用之前，你不可能知道。**

所以你没办法在写代码之前想清楚设计。你只能先做一个最小的东西，让 AI 去用，观察它在哪里高效、在哪里卡住，然后调整。这不是敏捷开发的套话，这是被逼出来的——因为你的用户是一个你尚未完全理解的智能体。

我们的每一次"砍功能"都不是深思熟虑的设计决策，而是"AI 用了之后发现没用"的事实驱动。overview 没用不是我们设计时就知道的，是 AI 在混淆代码上跑了一遍才知道的。`--data-only` 不是我们提前想到的，是 AI 被 26% 的噪声淹没后才意识到需要的。

**一边开发一边完善，不是因为我们不够专业，而是因为这是面对未知用户的唯一正确做法。**

传统产品方法论的核心假设是"需求可以被预先定义"。AI 产品打破了这个假设。

原因很简单：

1. **你不知道 AI 能做什么、不能做什么，直到你试。** 我们以为 AI 需要函数行为摘要，结果 AI 看到原始数据自己就能推理。我们以为 AI 需要 overview 做导航，结果 AI 用 search 更快。这些你写 PRD 的时候想不到。

2. **AI 的能力在快速变化。** 你今天写的需求文档，三个月后大模型升级了，一半需求变成了"不需要工具做，AI 自己能做"。需求文档的保质期太短了。

3. **交互模式从使用中涌现。** `--data-only` 这个 flag 不是设计出来的，是 AI 实际跑了一次 taint 看到 26% 噪声后才意识到需要的。`--range` 也是——分析 HMAC 区域时每次 search 全文太多结果，自然就需要限定范围。

4. **整条链路的问题只有跑起来才暴露。** unidbg 的 SIMD 寄存器 bug，我们设计阶段根本不知道这个问题存在。只有跑到 memdump 提取 HMAC key 时才发现值全是零，然后一路追到 Unicorn2 的 `uc_reg_read` 层面。

所以 AI 产品的正确做法是：

**不要写 PRD，写 Demo。不要开评审会，开终端跑一遍。不要讨论需求，让 AI 自己告诉你它需要什么。**

一个下午做出的能跑的 demo，比一周写出的完美设计文档有价值一百倍。因为 demo 能暴露真实问题，文档只能暴露你的想象力边界。

## 结语

回到最开始的问题：AI 时代，工具应该怎么设计？

**不要设计。先做，再改。**

传统产品方法论的核心假设是"需求可以被预先定义"。AI 产品打破了这个假设——你的用户是一个能力边界不断扩展的智能体，你没法提前画好它的使用路径。

唯一的办法是：给它一个最小工具集，让它去用，看它在哪里卡住，修那个点，再让它试。循环往复。

最终我们得到的 6 个命令，没有一个是"设计"出来的。它们是从十几次"AI 用不了→改→再试"的循环中**长出来的**。

把工具做笨，把决策留给 AI，把流程交给人。**然后去跑一遍，让现实告诉你下一步该改什么。**
